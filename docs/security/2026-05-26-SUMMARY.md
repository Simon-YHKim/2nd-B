## 🚨 CRITICAL (배포 차단 / 즉시 픽스 필요) — 3개

### C1. `users.subscription_tier` 자체 변조 가능
**출처**: security-checklist B
**파일**: `db/migrations/0009_rls_policies.sql:17` (`users_self_update`) + `0020_subscription.sql`
**문제**: `users_self_update` 정책 = `USING (id = auth.uid()) WITH CHECK (id = auth.uid())`. 컬럼 제한 없음.
**공격 시나리오**:
```js
// 인증된 사용자가 DevTools에서:
await supabase.from('users').update({ subscription_tier: 'brain' }).eq('id', user.id);
// → 즉시 Brain 등급 (Jarvis 일일 250회 + 모든 게이트 통과)
```
**영향**: 결제 우회 + 무제한 Gemini 사용
**픽스**:
- (A) Column-level GRANT: `REVOKE UPDATE (subscription_tier, subscription_expires_at, subscription_provider) ON users FROM authenticated;`
- (B) 트리거로 변경 차단:
```sql
CREATE FUNCTION block_tier_self_change() RETURNS trigger AS $$
BEGIN
  IF NEW.subscription_tier IS DISTINCT FROM OLD.subscription_tier
     AND auth.role() != 'service_role' THEN
    RAISE EXCEPTION 'subscription_tier modifiable by service_role only';
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
CREATE TRIGGER trg_block_tier ON users BEFORE UPDATE EXECUTE FUNCTION block_tier_self_change();
```
권장: (A) — RLS와 잘 결합.

### C2. `chat_usage` 자체 리셋 가능
**출처**: security-checklist C
**파일**: `db/migrations/0023_chat_usage.sql:30` (`chat_usage_owner_all`)
**문제**: `FOR ALL TO authenticated` 정책 = SELECT/INSERT/UPDATE/DELETE 모두 owner가 가능. 사용자가 자기 row를 UPDATE해서 count=0으로 리셋 가능.
**공격 시나리오**:
```js
await supabase.from('chat_usage').update({ count: 0 }).eq('user_id', user.id);
// 또는: await supabase.from('chat_usage').delete().eq('user_id', user.id);
// → 일일 카운터 리셋 → 무제한 Jarvis
```
**영향**: 일일 제한 우회 → Gemini 비용 폭탄
**픽스**: 정책을 SELECT만 owner에게, INSERT/UPDATE/DELETE는 service_role만으로 제한.
```sql
DROP POLICY chat_usage_owner_all ON chat_usage;
CREATE POLICY chat_usage_owner_select ON chat_usage
  FOR SELECT TO authenticated USING (user_id = auth.uid());
-- INSERT/UPDATE는 정책 없음 → SECURITY DEFINER인 bump_chat_usage()만 변경 가능
```
주의: bump_chat_usage RPC는 현재 SECURITY INVOKER. → `SECURITY DEFINER`로 바꾸고 search_path 고정 필요.

### C3. `gemini-proxy` 예산/길이 캡 없음 (paid-api-guard)
**출처**: paid-api-guard 6층 방어 + /cso Phase 7
**파일**: `supabase/functions/gemini-proxy/index.ts:79-91`
**문제**: 클라이언트 prompt 길이 제한 없음, `max_output_tokens` 없음, per-user/per-day 비용 캡 없음. C1+C2 조합 시 즉시 폭탄.
**공격 시나리오**:
1. tier='brain'으로 변조(C1) + chat_usage 리셋(C2)
2. `model='gemini-2.5-pro'` + `user`에 1MB+ 프롬프트
3. 무한 루프로 호출 → 분당 수십 달러
**픽스**: Edge Function에 4중 가드 추가:
```ts
const MAX_USER_LEN = 8000;   // chars
const MAX_SYSTEM_LEN = 4000;
if (userText.length > MAX_USER_LEN) return jsonResponse({ error: 'user_too_long', max: MAX_USER_LEN }, 413);
if (systemText && systemText.length > MAX_SYSTEM_LEN) return jsonResponse({ error: 'system_too_long' }, 413);
geminiBody.generationConfig = { maxOutputTokens: 1024, temperature: 0.7 };
```
+ 서버측 daily 비용 카운터 (별도 테이블 `gemini_spend_daily`, service_role만 write).
