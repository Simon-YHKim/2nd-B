# 2nd-Brain · 완벽 클론 프로토콜 (Claude Cowork — computer + chrome + android studio use)

> 목표: 정본(design/*.dc.html)의 **모든 디자인·상호작용·버튼·비율**을 실제 RN(Expo) 앱으로
> 픽셀에 가깝게 재현한다. 도구: chrome-use(정본을 띄워 본다) + android-use(에뮬레이터에서
> 결과를 본다) + computer-use(둘을 나란히 비교). "보고 → 만들고 → 나란히 대조 → 차이 0까지"의
> 루프가 핵심이다. 완료 정의가 전부 참이 될 때까지 멈추지 마라.

## 0. ⚠️ 치수 = 완전 반응형 (먼저 읽어라)
정본 .dc.html 의 폰 프레임은 **축소된 목업**이다(파일마다 300×700 / 320×680 / 390×844 로 제각각).
이 px 를 그대로 베끼지 마라. **어떤 화면 크기·비율에서도 자연스럽게 채워지는 반응형으로 만든다.**
세상의 안드로이드/iOS 는 폭 320~440dp, 화면비 16:9~20:9, 폴더블·태블릿까지 다양하다.

원칙:
- 정본은 "이 요소가 화면의 어디에, 어느 정도 비중으로, 무엇 옆에" 있는지의 **레이아웃 의도**로
  읽는다. 절대 px 가 아니라 그 의도를 재현한다.
- 레이아웃은 **flex + 비율(%) + gap**. 고정 px 폭 지양. 좌우 패딩 등은 화면폭 비례 또는 토큰 스케일.
- **세로는 스크롤 가능**하게(화면이 짧아도 잘리지 않게). 히어로/별자리처럼 큰 그래픽은 가용
  공간에 맞춰 스케일(aspect-ratio 유지). 콘텐츠가 길면 자연 스크롤.
- 타이포·간격은 **소수의 토큰 스케일**(예: sm/md/lg)로. 화면이 크면 살짝 키우고 작으면 줄이되
  단계는 토큰으로 통일. OS 폰트 스케일(접근성 큰 글씨) 존중 — 고정 px 폰트 지양.
- 안전영역(SafeArea insets)·노치·하단 제스처바를 반영해 어떤 기기에서도 가림 없이.
- 터치타깃은 화면 크기와 무관하게 **항상 ≥44pt**.
- 검증 기준 폭: 최소 **320dp**(작은 안드로이드) · 기준 **390dp** · 큰 **430dp** · 가능하면
  **600dp+(폴더블/태블릿)** 에서도 깨지지 않게(이땐 콘텐츠 최대폭 제한 + 가운데 정렬 허용).
- 390dp 는 "베끼는 크기"가 아니라 비율을 읽기 편한 **기준점**일 뿐이다.

## 1. 환경 준비 (Cowork)
- chrome-use: 로컬에서 정본을 연다. design/ 폴더를 간단한 정적 서버로 띄우거나 file:// 로
  각 .dc.html 을 연다. 브라우저 창을 390px 폭으로(devtools device toolbar, iPhone 12/13/14
  = 390×844) 맞춰 정본을 "실제 크기 비율"로 본다.
- android-use: 이미 띄워둔 가상 디바이스에 expo dev(npm run android 또는 expo start → a)로
  앱을 올린다. RN 핫리로드로 즉시 반영.
- computer-use: 왼쪽 Chrome(정본) · 오른쪽 에뮬레이터(구현)를 나란히 놓고 화면별로 대조.

## 2. 화면별 클론 루프 (SCREEN_TREE_SPEC.md 순서대로 전 화면)
각 화면에 대해:
1) **본다**: chrome-use 로 해당 정본 프레임을 연다(파일↔화면 매핑은 DESIGN_INDEX.md).
   레이아웃·계층·간격·폰트·색·아이콘·상태(빈/로딩/에러/채움)를 관찰.
2) **상호작용 캡처**: 정본/통합 프로토타입(2nd-Brain 프로토타입.dc.html)에서 버튼을 눌러
   "무엇을 누르면 무엇이 변하고 어디로 가는지"를 확인. 모션은 fade + glow-bloom, bounce 금지.
3) **만든다**: RN 화면을 deepSpace.* 토큰으로 구현(아래 토큰 표). 0번 비율 규칙 적용.
4) **나란히 대조**: 같은 화면을 에뮬레이터에서 띄워 Chrome 정본과 1:1 비교. 차이(간격·크기·
   색·정렬·폰트·아이콘·상태) 목록화 → 0 이 될 때까지 수정.
5) **상호작용 검증**: 에뮬레이터에서 실제 탭/입력/이동이 정본과 동일하게 동작하는지 확인.

## 3. 디자인 토큰 (정본에서 추출 — RN 은 @/theme/tokens 의 deepSpace.* 사용, hex 금지)

- 시안: #46B6FF / soft #9FE4FF / bright #CCFAFF / body #5FD4FF / receding #7FC9F0
- 텍스트: #E8F7FF(강조) · rgba(159,228,255,.6~.8)(muted)
- 소울 보라: #C8B6FF · #A78BFA · #8B7BD8 (북극성/AI 전용, 절제)
- CTA 민트: #5FF0C0 / 글자 #04241C  · 경고 #FFC478
- 카드: fill rgba(70,182,255,.06) · border rgba(70,182,255,.24)
- 폰트: 본문 Pretendard / 픽셀 제목 Galmuri11 / 라벨 Press Start 2P(영문 대문자)
- radius: 카드 14~16 · 폰 프레임은 실기기라 OS가 처리(목업의 34/38 은 무시)
- 금지: glassmorphism · pill 칩 · 이모지(thin SVG 라인만) · em dash · bounce/elastic · off-palette
- 캐릭터: 큰 머리(≥80) 터치 추적+둥둥+감정오브 / 작은 헤더 머리 추적 없음. 오브색 긍정 민트·중립 보라·부정 clay.

## 4. 상호작용 정본 (글보다 동작으로)
- **단일 진실원 = 2nd-Brain 프로토타입.dc.html**. 모션·전이·버튼 반응이 애매하면 이 파일을
  chrome-use 로 열어 직접 눌러보고 그대로 재현한다.
- 공통: 머리 터치 추적, 별 탭 시 머리가 그 별을 보고 말풍선 전환, 감정 오브 디밍, 화면 전환
  fade, 로딩은 DeepSpaceLoader, 긴 작업은 백그라운드 도크+완료 토스트(자동 이동 없음),
  propose→ratify(자동 반영 없음). 제스처/모션 디테일은 정본의 느낌을 따르되 RN 관용(Reanimated)
  으로, bounce 금지·fade+glow-bloom 만.

## 5. iOS 대비 (지금은 Android, 나중에 iOS)
- 절대 px·Android 전용 API 하드코딩 금지. SafeAreaInsets(상단 노치·하단 홈 인디케이터),
  플랫폼 분기(Platform.select)로 폰트/그림자/햅틱 차이 흡수.
- 상단 상태헤더는 insets.top, 하단 탭/도크는 insets.bottom 반영. 고정 9:41 목업 상태바는
  실기기 OS 상태바로 대체(목업 그대로 그리지 마라).
- 폰트 3종이 iOS 에서도 로드되는지(expo-font) 확인. 그림자: iOS shadow* / Android elevation 둘 다.

## 6. 자가검증 루프
화면 하나 끝낼 때마다: (a) Chrome 정본 ↔ 에뮬레이터 나란히 스크린샷, 레이아웃 의도(배치·비중·순서) 일치
(b) 실제 탭/입력/이동이 프로토타입과 동일 (c) npm run type-check 통과 + 그 파일 hex·em dash·
금지 마커 0 (d) **320 / 390 / 430 / 600dp+ 폭과 짧은·긴 화면비에서 깨짐·잘림·빈공간 없음**.
