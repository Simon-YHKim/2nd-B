05데이터 모델 변경 후보

옵션요약장점단점
A. 마이그레이션 (정통) sources 테이블에 wiki_track ENUM('daily','pro') 추가, SourceKind 에 note · email · image_ocr · file_doc 추가 쿼리 단순. RLS 분리 가능. 통계 정확. DB 마이그레이션 + 타입 회귀 다수.
B. tag 기반 (저비용) ⭐ frontmatter track: 'daily'|'pro' + 기존 tags 에 'pro' 추가. 새 kind 는 tag 'mode-ocr' 등으로 마이그레이션 불필요. 즉시 배포 가능. 쿼리/필터에 tag 의존 — 살짝 fragile.
C. 하이브리드 이번 PR = B (tag). 사용자 데이터 누적 후 v2 마이그레이션 (A) 위험 분산. 점진 진화. 두 단계 작업.

06제안 MVP (이번 PR 범위)

  • ✅ 일상 Wiki / Pro Wiki 토글 (tag 기반, 옵션 B)
  • ✅ 5 모드 탭 UI — 메모 / 링크 / 스크랩 / 캡쳐 OCR (placeholder) / 문서 (placeholder)
  • ✅ 메모 / 링크 / 스크랩 모드 = 기존 로직 reuse, 깔끔하게 분리
  • ⏳ 캡쳐 OCR / 문서 모드 = "곧 추가" placeholder — 외부 의존성 (`expo-image-picker`, `expo-document-picker`, OCR API) 필요. 다음 PR.
  • ✅ 톤 일관화 — "자재 반입" / "어떻게 들어왔어요?" / "영차영차 던지기" 식 카피
  • ✅ back 버튼 → router.push("/") (네비게이터 복귀)
  • ⏳ 자료별 읽기/수정 화면 — /wiki 가 이미 존재, 다음 PR 에서 모드별 필터 추가

07핵심 결정 (사용자에게)

Q1 · 데이터 모델

마이그레이션 vs tag 기반?

옵션 A (DB 마이그레이션) 가 정통이지만 작업 큼. 옵션 B (tag 기반) 는 이번 PR 안에 다 끝남. 옵션 C 는 둘 다. 제안: B 로 시작, 사용자 데이터 누적되면 A 로 마이그레이션.

Q2 · OCR/파일 범위

이번 PR 에 OCR/파일 입력도?

외부 의존성 (expo-image-picker, expo-document-picker, OCR API) 필요. 약 4시간 + 권한 처리. 제안: 이번 PR 은 placeholder ("곧 추가") 만, 다음 PR 에서 구현.

Q3 · Pro/일상 분류 방식

사용자 토글 vs AI 자동?

입력 시점 사용자 명시 토글이 가장 정확. AI 자동 분류는 v2 이후 (분류기 학습 데이터 누적 후). 제안: 사용자 토글, 기본값 = 일상 Wiki.
