# 2nd-Brain Kids · 유아 LLM 리터러시 트랙 (Concept)

> **AI를 "쓰는 법"이 아니라, AI가 "무엇인지"부터.**

**상태**: Concept / Phase-2 비전 트랙. **해커톤 범위 밖.** 출시 연령 floor는 14세 유지 (C10 불변).
이 문서는 방향 설계이며, 코드·스키마·제약 변경을 수반하지 않는다.
**관계**: `docs/VISION.md` 평생 아크의 *가장 앞단 on-ramp*. 핵심 제품(14+, 자기지식 축적)의 네 번째 축이 아니라 그 입구.
**작성**: Simon · 2026-06-01

---

## 1. 왜 존재하는가 — 평생 아크

핵심 제품은 14세 이상이 *자기 자신을 데이터로 축적*하는 도구다. 그런데 **LLM 리터러시는 14세보다 훨씬 일찍 시작돼야 한다.** 문제는, 텍스트 자기성찰 저널이 글 못 읽는 아이에게는 작동하지 않는다는 것.

이 트랙이 그 간극을 메운다:

- **4-7세**: 부모와 함께, 음성·그림으로 *"AI란 무엇인가"*를 체득 → 리터러시 on-ramp
- **8-13세**: (미래) 점진적 자기주도 확장
- **14세+**: 핵심 2nd-Brain으로 graduate — 도구를 *이해한 채로* 자기지식을 쌓기 시작

> 4살에 "AI는 살아있지 않은 기계"를 배운 아이가, 14살에 자기 2nd-brain을 짓는다.
> 이 평생 아크가 XPRIZE *Education & Human Potential* 트랙의 핵심 서사.

## 2. 누가 쓰는가 — 부모-운영 모델

- **계정 주인 = 부모(성인).** 일반 성인 가입·동의 경로를 그대로 탄다.
- **아이 = 감독 하 참여자. 별도 계정 없음. PII 수집 0.**
- 이 구조가 컴플라이언스의 핵심 — *아동에게서 개인정보를 수집하지 않으므로* COPPA / PIPA §22-2의 아동-동의 머신리(= PR-4)가 **불필요**해진다. 동의의 주체는 성인 부모.

대상 연령대는 현실적으로 **4-7세**. 3세는 관찰·동석 수준(상호작용 기대 낮음)으로만 둔다 — 솔직하게.

---

## 3. 발달 사다리 — 나이별로 잡히는 모델

| 나이 | 잡히는 멘탈 모델 |
|---|---|
| 4-5세 | "말하는 기계다. 살아있지 않다. 친구가 아니라 도구다." |
| 6-7세 | "예시를 많이 보고 배웠다. 그래서 가끔 틀린다." |

추상 개념·비판적 검증은 그 위 연령으로 올라가며 확장.

---

## 4. 무엇을 가르치는가 — 5개 리터러시 목표

| # | 멘탈 모델 | 왜 중요한가 |
|---|---|---|
| 1 | AI는 사람·친구가 아니라 **기계·도구** | 의인화 방지 — 유아의 AI 애착·과신 우려에 정면 대응 |
| 2 | AI는 **예시로 배운다** | ML 직관의 씨앗 |
| 3 | AI는 **틀릴 수 있다 / 지어낸다** → 어른에게 확인 | 환각 리터러시 (연령 맞춤) |
| 4 | **내가 정한다 / 끌 수 있다** | 도구에 대한 주도권 |
| 5 | 어떤 건 **기계가 아니라 사람에게** 묻는다 | 한계 + 안전 |

각 목표는 §5의 활동 하나로 체화된다.

## 5. 어떻게 — 제품·UX 모양

- **공동놀이(co-play)**: 부모 + 아이 함께. 한 세션 3-5분. 부모가 읽어주고 조작.
- **음성 입력 / 시각 출력**: 말하거나 그림 탭. **타이핑·읽기 불필요.**
- **일부러 사람 같지 않은 AI 캐릭터**: 의인화 방지를 *디자인 원칙*으로 (목표 #1 강화). 눈·표정 과잉 금지.
- **"생각하는 / 틀리는 걸 보여주는" 활동**: 예 — AI가 아이 그림을 보고 동물을 맞히다 *가끔 틀린다* → 학습·오류를 눈으로 봄 (목표 #2·#3 동시).
- **현실로 내보내는 off-ramp**: 끝나면 "이제 나가서 진짜 강아지 그려볼까" — 체류 유도 금지.

## 6. 데이터 · 컴플라이언스 아키텍처 (핵심)

| 원칙 | 내용 |
|---|---|
| 아동 PII 0 | 아이 계정·이름(선택 시 기기 로컬만)·생년월일·행동 프로파일 **저장 안 함** |
| 휘발성 입력 | 아이 음성은 *일시 처리 후 폐기*, 신원에 연결 안 함. 가능하면 on-device |
| 동의 주체 = 부모 | 성인 계정의 일반 동의로 커버 → 아동-동의 절차 불요 |
| 광고·서드파티 0 | Kids 표면에 광고·서드파티 분석 SDK 금지 (Apple Kids / Google Designed-for-Families 요건) |
| 부모 투명성 | 부모가 무엇을 다뤘는지 보고/듣기 가능 |

LLM 경로는 핵심 제약을 그대로 지킨다: **C1**(`gemini.ts` 단일 경로), **C3**(`ai_audit_log`).
단 **C9 분류기는 Kids 프로파일이 새로 필요** — 유아의 red zone은 성인의 "텍스트 자해 신호"가 아니라 *"유아 부적절·위험 생성물"*이라 위협면이 완전히 다르다. `src/lib/safety/lexicon.ts`에 kids 레이어를 더한다.

어휘 정책(blueprint §3)도 유지 — *"리터러시·놀이·배움"*이지 임상 용어 아님.

## 7. 재사용 vs 신규 (build delta)

| 재사용 | 신규 |
|---|---|

| Supabase auth (부모 = 성인 계정) | 부모-운영 세션 모델 + 휘발성 무저장 경로 |
| | 부모 대시보드 |

## 8. 패키징 옵션 (결정 보류)

- **(A) 앱 내 모드** — 부모 성인 계정 안의 "Kids" 섹션. auth·인프라 공유, 오버헤드 낮음. ← 프로토타입 권장
- **(B) 별도 앱** — Kids 카테고리 독립 등록. 컴플라이언스 경계 깔끔하나 오버헤드 큼. 스토어 출시 시 재검토.

## 9. 범위 경계 (명시)

- **Phase-2 비전 트랙.** 해커톤(마감 2026-08-17)은 **14세 floor만** 출시 (C10 불변).
- 이 문서로부터 **코드·스키마·제약 변경은 일절 착지하지 않는다.**
- 구현 약속이 아니라 *방향 고정*.
