## HIGH

### [ ] [edge-functions] `supabase/functions/gemini-proxy/index.ts`:120-132 — No per-user/per-day spend cap in proxy; daily limit is client-only and bypassable
*security · confidence 9*

The proxy enforces only per-call caps (MAX_USER_LEN, MAX_SYSTEM_LEN, maxOutputTokens) and checks that an Authorization header is present (startsWith 'bearer ') but never decodes the JWT, never derives the user id, and never consults chat_usage. The documented C3 remediation (docs/security/2026-05-26-SUMMARY.md item C3) called for a server-side per-user/per-day budget table (gemini_spend_daily, service_role-write) — grep shows it was never implemented (no 'gemini_spend' anywhere). The daily Jarvis limit is enforced via the client / bump_chat_usage RPC, not in the proxy. Result: any authenticated user (including a free/unlimited judge account, or anyone replaying a captured JWT) can call gemini-proxy in a loop with full-length gemini-2.5-pro prompts and bypass the daily cap entirely — a direct paid-API cost-amplification path that the project's $0/mo promise and paid-api-guard explicitly try to prevent.

**Fix:** In the proxy, decode the verified JWT to get user_id, then enforce the daily quota server-side: either call bump_chat_usage (SECURITY DEFINER) and reject with 429 when over limit, or maintain a gemini_spend_daily counter (service_role-write) and short-circuit before the upstream fetch. Do this before the fetch so the cap is authoritative regardless of which client calls.

### [ ] [db-migrations] `db/migrations/0025_security_hardening.sql`:37-63 (interaction with 0019 award_xp lines 65-98) — block_self_tier_change trigger breaks award_xp for every authenticated user
*correctness · confidence 9*

0025 adds a BEFORE UPDATE trigger trg_users_block_self_tier on public.users that RAISES 42501 whenever total_xp (and 5 other columns) IS DISTINCT FROM OLD, unless current_setting('request.jwt.claim.role', true) = 'service_role'. The only sanctioned XP path, public.award_xp (0019), does `UPDATE public.users SET total_xp = total_xp + v_rule.xp_value`. Although award_xp is SECURITY DEFINER, the trigger still fires on the UPDATE, and SECURITY DEFINER does NOT change the PostgREST request.jwt.* GUCs — for an authenticated RPC caller that setting is 'authenticated' (or NULL on PostgREST v9+, which dropped the singular per-claim GUCs). It is never 'service_role'. So the guard's early-return is never taken, NEW.total_xp differs from OLD.total_xp, and the whole award_xp transaction aborts with 'protected user column may only be changed by service_role'. Result: XP/leveling is completely broken in production. The client wrapper awardXpSafe() (src/lib/progression/xp.ts) swallows the error to a console.warn, so the failure is silent and the onboarding quest / levels never advance. CI never caught it because supabase-dry-run only applies the migrations to vanilla Postgres and never invokes award_xp under an authenticated JWT, and there is no SQL test for the trigger/RPC interaction.

**Fix:** Exempt the trusted definer RPC from the guard. Either (a) move total_xp / onboarding_quest_completed_at out of block_self_tier_change's protected set (they are already only writable via the SECURITY DEFINER award_xp, which derives the amount server-side from xp_rules and never trusts the client), or (b) have award_xp set a transaction-local marker the trigger honors, e.g. `PERFORM set_config('app.allow_xp_write','1',true);` inside award_xp and add `IF current_setting('app.allow_xp_write', true) = '1' THEN RETURN NEW; END IF;` at the top of block_self_tier_change. Add a behavioral SQL test (apply migrations, SET request.jwt.claims to an authenticated user, call award_xp, assert success) so the dry-run gate covers it.

### [ ] [data] `src/lib/records/delete-bulk.ts`:108-121 — deleteAllUserData is an incomplete erasure — most user-owned tables survive a "full wipe"
*security · confidence 9*

deleteAllUserData() only deletes from records, sources, wiki_pages, and chat_usage. It never deletes the users row. Both callers present this as terminal erasure: src/app/account.tsx onDeleteAccount() ('Delete account' → deleteAllUserData + signOut) and src/app/settings.tsx runFullWipe() ('Full wipe complete'). Many other tables are owned by the user and reference users(id) ON DELETE CASCADE: personas (0008), memorized_patterns (0017), xp_events (0019), self_contexts (0021), clipper_templates.owner_id (0027), consent_records (0031), testimonials (0006). Because the users row is never deleted, none of these cascade — so after a 'full wipe' the user's inferred persona snapshots, memorized journal-derived patterns, XP history, self-context entries, clipper templates, and testimonials all persist. This directly contradicts the documented erasure design: docs/research/methodology/ai-memory-architecture.md §7 states 'User-initiated deletion is the only path to physical erasure' and 'A user deletion is a single DELETE … CASCADE across all tiers when everything lives in one Postgres.' memorized_patterns in particular stores distilled user journal text (see create.ts), so this is residual PII after the user asked to be erased — a GDPR Art.17 / PIPA erasure gap. account.ts even acknowledges 'removing the auth.users record itself needs a service_role path … handled out of band,' but that out-of-band path does not exist, so nothing ever triggers the cascade.

**Fix:** Implement true erasure: add a service_role edge function (or RPC) that deletes the auth.users row so the ON DELETE CASCADE fans out to every owning table, and have deleteAllUserData (or the account-delete flow) invoke it. As an interim client-side measure, explicitly delete the remaining owned tables (personas, memorized_patterns, xp_events, self_contexts, clipper_templates, consent_records, testimonials) in dependency order before signOut, and update the returned count shape/UI to reflect them. Add a test asserting every table that REFERENCES users(id) is covered.

### [ ] [auth] `src/lib/supabase/consent.ts`:78-88 (and callers src/app/(auth)/sign-up.tsx:73, complete-profile.tsx:65) — Consent-ledger write is fire-and-forget and silently swallows failures (PIPA accountability gap)
*constraint · confidence 8*

recordConsentBestEffort() catches every error and returns false with only a console.warn, and both callers invoke it via `void recordConsentBestEffort(...)` immediately before `router.replace('/')`. The consent_records table (migration 0031) is the project's PIPA accountability record of WHAT the user agreed to and under which policy versions (the file header and 0031 header both state this purpose explicitly). If the INSERT fails for any reason other than a missing table — RLS hiccup, transient network error, or the navigation cancelling the in-flight request on web — the account is created and the user proceeds with NO durable proof of consent, no retry, and no telemetry. The UI submit gate (allRequiredAcksChecked) is the real consent moment so users are not let in without consenting, but the legal audit trail required for C10/PIPA can be lost without anyone noticing. Impact: under audit, some accounts may have no consent_records row despite the constraint claiming an immutable ledger.

**Fix:** Distinguish 'table not yet migrated' (acceptable no-op) from a genuine write failure, and on genuine failure either (a) retry with backoff, (b) await the write before navigating away so a failure surfaces, or (c) enqueue it durably (e.g. persist to AsyncStorage and flush later) and emit a metric/Sentry breadcrumb. At minimum, await the promise instead of `void`-ing it so a hard navigation doesn't cancel it, and log failures at error (not warn) level so they are visible in monitoring.

### [ ] [edge-functions] `supabase/functions/gemini-proxy/index.ts`:23, 237-241 — C3 violated: gemini-proxy makes a Gemini call but never writes ai_audit_log
*constraint · confidence 8*

The proxy forwards the prompt to Gemini and returns the text, but it never inserts an ai_audit_log row (confirmed: no insertAiAuditLog / ai_audit_log reference anywhere under supabase/functions). The header comment states 'Audit-log writes still happen on the client after the reply lands' — i.e. the only C3 INSERT lives in src/lib/llm/gemini.ts (lines 268-285), which runs in the React client. C3 requires an audit row on EVERY Gemini call. ### [ ] [edge-functions] `supabase/functions/oauth-naver/index.ts`:121-154 — Naver OAuth find-or-create matches by email, enabling cross-provider account linking/takeover
*security · confidence 8*

The function locates an existing account purely by email equality (u.email === userEmail) and, when found, mints a magic-link session for that pre-existing userId. Because email is the only join key, a Naver login whose profile email equals an account previously created via email/password, Google, or Kakao silently grants the Naver flow full access to that other account. If any provider in the system allows an unverified or attacker-chosen email (or if a user changes their Naver-account email to a victim's address), this is an account-takeover vector. Additionally, when Naver does not return an email (email scope not granted) the code falls back to `${naverId}@naver.local`; that synthetic address is fine for isolation, but the real-email branch is the takeover risk. createUser also sets email_confirm:true unconditionally.

**Fix:** Do not link accounts on email alone. Match on a provider-stable identity key (store naver_id and look up by it, e.g. via user_metadata or a dedicated oauth_identities table) rather than email. If an email collision with a different provider is detected, refuse to auto-link and surface an explicit account-linking flow instead of minting a session. Only treat the email as authoritative when the existing account's email was itself verified by the same trust level.

### [ ] [db-migrations] `db/migrations/0011_security_fixes.sql`:16-18 — Client-writable ai_audit_log INSERT policy lets the audit trail (C3) be forged or skipped
*constraint · confidence 9*

0011 grants authenticated users a direct INSERT on ai_audit_log with only `WITH CHECK (user_id = auth.uid())` and no validation of prompt_hash/output_hash/model_used/vertex_backend/safety_zone. ### [ ] [persona] `src/lib/journal/streak.ts`:11-19 — KST streak day-boundary is wrong on any non-UTC device (breaks for Korean users)
*correctness · confidence 9*

kstDayKey computes `const utc = new Date(iso).getTime() + now.getTimezoneOffset() * 60_000;` then `new Date(utc + KST_OFFSET_MS)`. `new Date(iso).getTime()` is already an absolute (UTC) epoch, so adding the runtime's `getTimezoneOffset()` is spurious. On a UTC host (CI) the offset is 0 and the +9h shift yields the correct KST wall-clock day, which is why the jest suite (now = '2026-05-25T12:00:00Z', run under UTC) passes. But on the real audience's devices the offset cancels or compounds the +9h: a KST device (UTC+9) has getTimezoneOffset() = -540, so utc = epoch - 9h, then +9h = epoch -> the function returns the UTC calendar day, NOT the KST day. A capture at 2026-05-25 07:00 KST (= 2026-05-24 22:00 UTC) is then bucketed under 2026-05-24. Consequence: the daily-capture streak shown in capture.tsx (line 212/759) is computed against the wrong day boundary for every user not in UTC, so streaks miscount, break a day early, or merge across the KST midnight the code claims to anchor on. This is a user-visible correctness bug in a Korean-first app, masked by UTC-only tests.

**Fix:** Drop the timezone-offset term entirely and shift the absolute epoch directly: `const kst = new Date(new Date(iso).getTime() + KST_OFFSET_MS);` then read getUTCFullYear/Month/Date as today. Remove the unused `now` argument from kstDayKey (todayKst already passes now.toISOString()). Add a regression test that runs with a mocked non-UTC TZ (e.g. process.env.TZ='Asia/Seoul') asserting a 07:00 KST timestamp lands on the KST day.
